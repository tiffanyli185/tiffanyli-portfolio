Overview

This project is the final lab of ECE 2300, building a full TinyRV1 RISC-V computer system on a DE0-CV FPGA: a single-cycle processor, a specialized accumulate accelerator, and an embedded I/O system, all wired together and running real assembly programs, including a door monitor, a two-function calculator, and an array-accumulate loop. It's the capstone of a three-lab progression, each one building the hardware and design skills the next one depended on.

Lab 2 built a two-function calculator, introducing combinational logic building blocks — an ALU, muxes, and basic control signals — as a self-contained, single-purpose circuit. Lab 3 built a multi-note music player, adding sequential logic, finite-state machines, and memory-mapped I/O to that foundation. Lab 4 combines all of it: the ALU and control logic from Lab 2, the FSM and memory-mapped I/O patterns from Lab 3, and a full RISC-V instruction set, register file, and datapath on top, wired to real FPGA peripherals (seven-segment displays, a distance sensor, a multi-note player, and an SPI program loader).

System Architecture

The processor was built up part by part rather than all at once: the immediate generator, ALU, register file, PC logic, and control signals were each implemented and checked individually against the instruction control-signal table from lecture before being wired into a single datapath and control unit. That datapath then connects to a memory bus, which arbitrates between an SPI loader (for programs sent from a host workstation) and the physical memory and external I/O devices the processor reads and writes through memory-mapped addresses.

The highlighted critical path corresponds to a load-word instruction: from the control unit, through the op2_sel signal, into the ALU adder, back through the memory bus, into dmem_rdata, through the writeback mux, and into the register file. This matches what's expected from lecture, since load word touches nearly every stage of the datapath in one cycle.

Accumulate Accelerator

Alongside the general-purpose processor, we designed a specialized accumulate accelerator that sums an array of 32-bit values directly in hardware, without running any instructions. Its datapath needs a counter to step through array indices, a multiplier to convert each index into a byte address, a memory interface to fetch one element per cycle, and an adder plus register to hold the running sum, with muxes selecting each of these based on the current state.

The three-state FSM (BEGIN, ACCUM, DONE) exists because the valid/ready handshake protocol requires separate phases for accepting a new request, doing the accumulation work, and holding the finished result — without them, the accelerator could reset mid-computation or violate the rule that in_rdy can't depend on in_val. BEGIN latches the array size and issues a single-cycle reset; ACCUM performs one memory read and one add per cycle until the counter signals done; DONE holds the result stable and re-asserts in_rdy until the next request arrives.

Area & Timing Analysis

Quartus reported the processor uses 6,244 ALMs, 9,072 ALUTs, and 5,755 dedicated logic registers — about 34% of the DE0-CV's fabric. That's a large jump from earlier labs: the Lab 2 calculator used only a few hundred ALUTs, and the Lab 3 music player used roughly 300–500, so the processor is 20–30x larger than either. That's the direct cost of generality — a full instruction set, 32-register file, branch/jump logic, and a shared datapath capable of executing many different instructions, versus task-specific hardware built for exactly one job.

On timing, the processor met its 80ns cycle constraint with a critical path of 63.280ns. Using time/program = instructions/program × cycles/instruction × time/cycle, accumulating 31 elements takes 159 cycles (5 instructions per loop iteration × 31, plus 4 setup instructions) at 1 cycle/instruction, for an expected execution time of 159 × 80ns = 12,720ns — which matched the 12.7µs period measured directly on the oscilloscope.

Testing & Results

Each stage was checked against hardware evidence rather than just simulation. The accumulate assembly program below was the workload used throughout the area and timing analysis: it initializes the loop counter, running total, and data pointer, then loads, adds, and advances the pointer each iteration until the counter reaches zero.

The accelerator was cross-checked the same way: its interactive simulator produced 159 "stat" cycles (the cycles where out3 is high, i.e. actual fetch-and-add work) for a size-31 array — matching the processor's own 159-cycle count for the equivalent software loop, with about 4 extra cycles from the final branch fall-through and cleanup before the store.
